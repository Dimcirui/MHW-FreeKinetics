# MHW Free Hyper-Kinetics — Blender 2.79 → 4.x 移植计划

> 交接文档。执行会话（Sonnet）请通读本文件后再动手。
> 本计划基于对当前 2.79 `main` 与一次失败的 3.1 移植尝试
> (`github.com/Dimcirui/MHW-Free-HyperKinetics-for-Blender31`) 的对比取证，
> 以及对核心解析层的独立运行验证。

---

## 1. 目标与范围

- **主目标**：在 **Blender 4.3** 上可用（与 4.2 LTS 同属 4.x API 代际，做好 4.3 通常也覆盖 4.2）。
- **跨版本兼容优先级**（仅在不增加显著成本时追求）：**4.x > 5.0 > 3.x**。
- **明确不支持**：5.1 及以上。
- **基线策略**：从 **2.79 `main` 干净重做**，每步保持**最小可审查 diff**。
  把上面那个 3.1 port 仅当"答案参考"，**不要**在它上面修补
  （它被 PEP8 重排污染、无法再与上游对照验证逻辑，且转换不彻底）。

## 2. 已验证事实（降低不确定性）

### 2.1 核心解析层完全不依赖 Blender —— 已实测通过 ✅
`struct/` 与 `common/`（除 `struct/freedomUniteAnim.py` 外）不 `import bpy`。
用一个极简 `mathutils.Vector` stub（Blender 本身自带 mathutils）即可在纯 CPython 下解析真实 `.lmt`。
**结论：这一层（.lmt/.timl/.efx 二进制格式、插值算法）移植时零改动。**

实测基线（`parseLMT` 独立运行，见 §7 测试台）：

| 样本 | version | entryCount | 解析出 ActionHeaders | frames | fcurveCount/bones | 备注 |
|---|---|---|---|---|---|---|
| `em023_07_fs.lmt` (≈3.6KB) | 95 | 68 | 10 | **1** | 8 | 每个 action 均带 TIML，单帧姿态 + TIML 驱动 |
| `em023_07.lmt` (≈3.4MB) | 95 | 274 | 123 | 36 / 34 | 131 | 常规骨骼动画，均带 TIML |

> 推论：`_fs` 类动画以 **TIML** 为主驱动。这正是下面 §4 头号缺陷（TIML 属性未注册）的命中区，
> 也是上一次移植"序列帧动画出问题"的最可能根因。

### 2.3 最难的迁移区域本项目【不存在】

- fcurve / action / keyframe API 跨 2.79→5.x **稳定**，核心导入逻辑不需改写。

## 4. 上一次 3.1 移植的【确认缺陷】——执行时务必避免重蹈

> 它是 LLM 机械移植：大量 PEP8 重排 + 选择性 API 改写，转换不彻底。下列为实锤问题：

1. **🔴 [timl_controller.py](blender/timl_controller.py) 的 797 个属性未转注解**（仍 `mXXXX = FloatProperty(...)`）。
   `TIMLController(bpy.types.PropertyGroup)` 经 `register_class` 注册并挂为 `Object.FreeHKTiml`，
   但旧语法属性在 3.x+ 被**静默跳过不注册** → **所有 TIML 时间线参数实际不存在**。
   文件头注明 "Automatically generated by PropGenerator.py" → 最佳修法：改生成器输出注解版，或脚本化批量转换（注意 **Tab 缩进 + 多行定义**）。**头号根因。**
2. **🟠 转换不彻底**：`operators/timl_io.py`（28 转、剩 4 个 `=`）、`operators/export_ops.py`（剩 1 个 `=`）。
3. **🟠 矩阵 `@` 漏一处**：[operators/lmt_rig_ops.py:419](operators/lmt_rig_ops.py)
   `(orthogonal.matrix*Vector((0,1,0,0)))` 未改 → 该 rig operator 运行即 `TypeError`。
   （注：[blender/tetherOps.py](blender/tetherOps.py) 的 4 处 `@` 转换是**正确**的，可照搬。）
4. **🟡 `scene.update()` 被直接删除**而非替换（[blender/binary_inputs.py](blender/binary_inputs.py)、[blender/pl_tools.py](blender/pl_tools.py) 内联处）→
   导入后骨架/tether 姿态可能不刷新，表现为"数据对、视口显示错"。
   （注册 handler 处改用 `depsgraph_update_post` 是对的，可照搬。）
5. **🟡 节点 socket `.new()` 改写不一致**：仅一处把第三位置参数改成 `identifier=`，其余几十处同样调用未动。统一处理。

## 7. 独立解析测试台（已验证可复现）
不依赖 Blender，用于 Phase 1 / 回归。放置于 `C:\tmp\fhkparse\`（执行会话可重建，或落入仓库 `tests/`）。

`mathutils.py`（stub）：
```python
class Vector(list):
    def __init__(self, seq=()): super().__init__(float(v) for v in seq)
    @property
    def x(self): return self[0]
    @property
    def y(self): return self[1]
    @property
    def z(self): return self[2]
    @property
    def w(self): return self[3]
    def __mul__(self, o): return Vector([a*o for a in self])   # scalar
    __rmul__ = __mul__
    def __add__(self, o): return Vector([a+b for a,b in zip(self,o)])
```
布局：把仓库 `struct/`、`common/` 复制到包目录 `fhk/`（含空 `fhk/__init__.py`），
将 stub 与 `fhk/` 同置于 `sys.path`，然后：
```python
import sys; sys.path.insert(0, r"C:\tmp\fhkparse")
from fhk.struct.Lmt import parseLMT
lmt = parseLMT(path)   # lmt.Header / lmt.ActionHeaders[*].{id,frameCount,fcurveCount,fcurveOffset,timlOffset,bones}
```
> 注：包名用 `fhk` 以避免与 stdlib `struct` 冲突；Py3 绝对 import 下 `import struct`（LmtBuffers 内）仍取 stdlib，无冲突。

## 9. Runtime 修复日志（在 5.1 压测 + 4.3 实测中发现，原计划未覆盖的整类问题）

> 规律：原迁移工具每次只匹配某一种正则形式，**整类整类地漏**。注册一报错即中止、
> 后续模块从未执行，故采用"在 5.1（比 4.3 严格）压测找 4.x 真 bug + 主动全扫"的策略。
